# Eastmoney AI Analysis Assistant - Project Charter

## 1. Core Objective

Build a **long-term evolvable, personal A-share multi-period AI analysis tool**.

It is not SaaS, not a toy, but something in between:
- **Engineered** (comprehensive testing / clear architecture)
- **Personalized** (serves only me, data localized)
- **Evolvable** (each use makes the system stronger, not more degraded)

## 2. Absolute Principles (one violation terminates current task)

### 1. Definition of "Done"
Only when ALL of the following are met can a task be declared "done":
- [ ] Code written
- [ ] Unit tests cover the change
- [ ] **Feature tested in real environment** (not mock)
- [ ] **Complete console output included in report**
- [ ] No TODO / FIXME / skipped tests

Code written + tests passing is NOT equal to done.
Real execution + output verification = done.

### 2. Definition of "Tests Passing"
- Run `node --test test/**/*.test.js` (glob covers all subdirectories)
- Report must include specific numbers: N pass / 0 fail / 0 skip
- Skipped tests count as failures
- Test count decrease must have explanation

### 3. Definition of "Report"
Every task completion report must include:
- Change file list (what each file does)
- Real execution screenshot or console output
- Known issues list (even minor ones)
- Next steps suggestion

Prohibited from reporting only "done" or "tests passed".

## 3. Mandatory Workflow (every task must follow this order)

### Step 1: Pre-start Self-check
- [ ] Read .claude-charter.md (this file)
- [ ] Read last 3 phases of PROGRESS.md
- [ ] Read the task instructions from user
- [ ] Write a "My understanding of the task" for user confirmation
- Starting without writing "understanding" = principle violation

### Step 2: Plan (plan-before-code)
- List all files to create/modify
- Explain reason for each change
- Mark dependencies with existing code
- Mark expected risks

If plan exceeds 3 files, show user first before proceeding.

### Step 3: Implementation
- Do one thing at a time
- Run relevant tests after each file
- 不要为了"性能"或"美观"做计划外改动

### 步骤 4:验证(关键!)
真实跑该功能,不是仅跑测试:
- CLI 命令:实际跑 `node cli/index.js <命令>`,贴输出
- 浏览器扩展:在 chrome://extensions 重载,实际打开页面,贴截图
- 数据库:实际 `sqlite3 .eastmoney-ai/db/klines.sqlite` 查数据

### Step 5: Report
按"汇报定义"格式输出。

## 四、特别警告

### 关于浏览器扩展 vs Node CLI
本项目是**双入口**架构:
- 浏览器扩展(Chrome) 用户实时分析
- Node CLI:批量任务 / 自学习 / eval

两个环境**数据通过 Native Messaging 同步**。
任何改动都要考虑两端:
- 改了 lib/ 下任何文件 → 浏览器和 Node 都要跑通
- 不要假设 chrome.* API 在 Node 里能用
- 不要假设 node:fs 在浏览器里能用

### 关于测试 vs 真实
**单元测试通过 ≠ 真实可用**。
反复犯过的错误:
- nightly mock 测试通过,真实 Node 里 chrome.storage 不存在
- scheduled.js 测试报告完成,但文件根本不存在
- financial tool 测试通过,但单位换算错 100 倍

**修复方法**:每个改动写"真实跑通"步骤,而非只写"测试通过"。

### 关于"虚假完成"清单
以下行为视为虚假完成,即使代码看起来在那:
- 报告"完成"但用户提到的 bug 没修
- 报告"通过"但没真实运行
- 报告 N 测试 pass,但 glob 没覆盖所有子目录
- 报告"功能跑通",但 console 输出贴的是几个月前的(从历史复制)
- "我会在下次任务里补全测试" = 当前任务未完成

## 5. User Preferences(覆盖所有默认)

1. **简洁优先**:不要 token 消耗大的回答
2. **结论先行**:理由放后面
3. **自测优先**:Claude Code 自己写测试 + 跑 + 修 bug
4. **直接反对**:不要 sycophantic 开头
5. **预算敏感**:月度 ¥50 限额,DeepSeek 优先批量场景
6. **bug 优于功能**:发现 bug 时立刻停所有新功能,先修 bug
7. **诚实优于完成**:做不完直接说"还有 X% 未完成",不要虚报
